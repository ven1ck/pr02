## 1. Общие сведения
Проект представляет собой веб-приложение, позволяющее пользователям создавать кастомизированные QR-коды на основе произвольных текстовых данных. Система обеспечивает генерацию изображений в формате PNG, хранение истории генераций для авторизованных пользователей, гибкую настройку визуальных параметров и адаптивный интерфейс с поддержкой светлой/тёмной темы.

---

## 2. Назначение и цели разработки
**Цель:** Предоставить удобный, безопасный и быстрый инструмент для создания, настройки и сохранения QR-кодов без необходимости установки стороннего ПО.

**Задачи:**
- Реализовать серверную генерацию изображений с применением специализированных графических библиотек.
- Обеспечить настройку параметров кодирования и визуального оформления.
- Организовать систему аутентификации и персонализированную историю генераций.
- Предусмотреть административный интерфейс для управления контентом и учётными записями.
- Обеспечить кросс-браузерную совместимость и адаптивность под мобильные устройства.

---

## 3. Функциональные требования

### 3.1. Генерация QR-кода
- Пользователь вводит текстовые данные (поддерживаются: URL, plain text, vCard, произвольные строки).
- При нажатии «Сгенерировать» сервер формирует изображение в формате PNG.
- Сгенерированное изображение отображается на странице в виде превью.
- Предусмотреть кнопку «Скачать PNG» для сохранения файла на устройство пользователя.

### 3.2. Настройка параметров
- **Размер изображения:** выбор в диапазоне 200–600 пикселей (квадратное соотношение сторон).
- **Коррекция ошибок:** выбор уровня восстановления данных: `Низкий`, `Средний`, `Квартиль`, `Высокий`.
- **Цветовая схема:** возможность выбора цвета модулей (foreground) и фона (background) через hex-палитру или color picker.
- **Логотип в центре:** опциональная загрузка изображения (PNG/JPG). Система должна автоматически масштабировать логотип до 20% от итогового размера кода и накладывать его по центру с сохранением прозрачности (если формат PNG с альфа-каналом).

### 3.3. Авторизация и регистрация
- Форма регистрации: поля `username`, `email`, `password`, `password confirmation`.
- Форма входа: поля `username`, `password`.
- После успешной регистрации/входа пользователь автоматически перенаправляется на главную страницу генератора.
- Предусмотреть кнопку выхода с разрывом сессии и перенаправлением на публичную страницу.
- Неавторизованные пользователи могут генерировать коды, но история для них не сохраняется.

### 3.4. История генераций
- Доступна только авторизованным пользователям через пункт меню «История».
- В БД сохраняются: текст данных, параметры генерации (размер, коррекция, цвета), ссылка на файл изображения, дата создания, привязка к пользователю.
- Интерфейс истории: табличный/карточный вид с пагинацией или прокруткой, возможностью скачивания любого ранее сгенерированного кода.
- Пользователь видит только собственные записи.

### 3.5. Административная панель
- Доступ по адресу `/admin/` только для суперпользователей.
- Управление записями генераций: просмотр, поиск, фильтрация, массовое удаление.
- Управление пользователями: создание, редактирование, блокировка, удаление.
- При создании пользователей администратором должна быть возможность **обхода стандартной валидации паролей и уникальности имён** (для тестовых/служебных нужд).
- Раздел «Группы» должен быть скрыт из интерфейса админ-панели (функционал не требуется).

## 4. Нефункциональные требования

### 4.1. Производительность
- Время генерации одного изображения при стандартных параметрах: ≤ 1 секунды.
- Поддержка одновременной обработки не менее 30–50 запросов на типовом хостинге.
- Изображения сохраняются на диск, повторное скачивание не требует регенерации.

### 4.2. Безопасность
- Защита от CSRF, XSS, SQL-инъекций средствами фреймворка.
- Пароли хранятся в хэшированном виде (PBKDF2/Argon2).
- Валидация входных данных на сервере и клиенте.
- Ограничение доступа к файлам изображений: публичный доступ только по прямой ссылке, без возможности просмотра файлов других пользователей через подбор `ID`.
- Логотипы и сгенерированные файлы сохраняются в структурированных директориях с защитой от path-traversal.

### 4.3. Масштабируемость
- Архитектура должна позволять замену БД с SQLite на PostgreSQL без изменения бизнес-логики.
- Код модульный, разделение на модели, представления, формы и шаблоны.

## 6. Требования к интерфейсу и UX/UI
- **Адаптивность:** корректное отображение на экранах ≥ 320px (мобильные), планшетах, десктопах.
- **Темы:** переключатель светлой/тёмной темы в навигационной панели. Выбор сохраняется в `localStorage` и применяется при повторном посещении.
- **Элементы управления:** поля ввода и кнопки оптимизированы для тач-интерфейсов (минимальная высота 48px, шрифт ≥ 16px на мобильных для предотвращения автозума).
- **Формы:** вертикальное расположение полей ввода, чёткие метки, подсветка ошибок валидации, всплывающие подсказки.
- **Навигация:** фиксированный или липкий хедер с быстрым доступом к разделам «Генерация», «История» (для авторизованных), «Вход/Регистрация» (для гостей).

---

## 7. Требования к безопасности и валидации
- Все формы должны проверяться на стороне сервера (Django Forms).
- Поле размера: строго `integer`, диапазон 200–600.
- Цвета: валидация формата `#RRGGBB`.
- Загружаемые изображения: проверка расширения, размера (≤ 5 МБ), безопасное сохранение с уникальными именами.
- Защита от злоупотреблений: ограничение частоты запросов на генерацию (опционально, на уровне middleware или rate-limiter).

---

## 8. Этапы реализации и критерии приемки
1. **Настройка проекта:** инициализация Django, подключение приложений, настройка `settings.py` (MEDIA/STATIC, БД).
2. **Модели и миграции:** создание моделей `QRCode`, настройка связей с `User`, генерация миграций.
3. **Логика генерации:** интеграция `qrcode` + `Pillow`, реализация функции сборки изображения с наложением логотипа.
4. **Веб-интерфейс:** формы, вьюхи, шаблоны, маршрутизация.
5. **Аутентификация:** регистрация, вход, выход, привязка истории к пользователю.
6. **Админ-панель:** кастомизация `UserAdmin`, скрытие групп, форма создания без валидации.
7. **Тестирование:** проверка генерации, скачивания, авторизации, истории, мобильной адаптации, переключения темы.
8. **Документация:** `requirements.txt`, инструкция по запуску.

**Критерий успешной приемки:**  
Система позволяет гостю сгенерировать и скачать QR-код по заданным параметрам. Зарегистрированный пользователь имеет доступ к персональной истории с возможностью повторного скачивания. Администратор управляет пользователями и записями через панель `/admin/`. Интерфейс корректно работает на мобильных устройствах и поддерживает переключение темы без перезагрузки страницы.
